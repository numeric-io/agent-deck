    - Test: ResolveSkillCandidate finds skill by name from the correct source
    - Test: Pool skill directory structure (no scripts/) is handled without error

  <action>
Create `internal/session/skills_runtime_test.go` in the `session` package.

All tests use `setupSkillTestEnv(t)` for environment isolation and `writeSkillDir(t, ...)` for creating test skills.

1. `TestSkillRuntime_AttachedSkillIsReadable`: Set up test environment. Create a skill source with `writeSkillDir`. Register source via `SaveSkillSources`. Create a temp project directory. Call `AttachSkillToProject(projectPath, "my-skill", "local")`. Read the SKILL.md at the returned `attachment.Path + "/SKILL.md"` using `os.ReadFile`. Assert file is readable and contains the expected skill name.

2. `TestSkillRuntime_ApplyCreatesReadableSkills`: Set up test environment. Create 2 skills in a source. Resolve both via `ResolveSkillCandidate`. Call `ApplyProjectSkills(projectPath, candidates)`. Walk the `projectPath/.claude/skills/` directory. Assert 2 skill directories exist. For each, `os.ReadFile("SKILL.md")` and assert readable.

3. `TestSkillRuntime_DiscoveryFindsRegisteredSkills`: Set up test environment. Create source with 3 skills. Register source. Call `ListAvailableSkills()`. Assert result contains all 3 skills by name.

4. `TestSkillRuntime_ResolveByName`: Set up test environment. Create source "src1" with skill "alpha". Create source "src2" with skill "beta". Register both. Call `ResolveSkillCandidate("alpha", "src1")`. Assert no error, name == "alpha", source == "src1". Call `ResolveSkillCandidate("beta", "src2")`. Assert no error.

5. `TestSkillRuntime_PoolSkillWithoutScripts`: Create a skill directory with only SKILL.md (no scripts/ subdirectory, matching pool skill pattern like gsd-conductor). Attach to project. Assert no error. Assert SKILL.md readable. This verifies pool skills (which often have only references/, no scripts/) work correctly.

6. `TestSkillRuntime_ResolveSkillContent`: After attaching a skill, read its SKILL.md content and verify it contains the expected frontmatter (name, description). Parse YAML frontmatter between `---` delimiters. Assert the `name` field matches.

These tests go beyond catalog operations (attach/detach exist in `skills_catalog_test.go`) to verify runtime behavior: the skill content is actually accessible from the materialized path, which is what matters when Claude loads a skill via `Read ~/.agent-deck/skills/pool/<name>/SKILL.md`.
